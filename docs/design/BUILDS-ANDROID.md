# BUILDS-ANDROID — Inkternity Android build pipeline (design)

Companion to `BUILDS.md`. That doc covers the desktop release pipeline
(Linux Flatpak / Windows NSIS / macOS DMG via tag-triggered GitHub
Actions, soft-fail signing). This one covers what's needed to add
Android (APK/AAB) to that pipeline.

Status: design for review. Not yet wired into CI; no Android job in
`build-installers.yml`.

---

## TL;DR

- An SDL3 Android scaffold already exists at `android-project/`. It's
  ~70% wired but has **never been built** in CI and was last touched
  pre-rebrand (still uses the `com.erroratline0.infinipaint` namespace).
- Three concrete blockers between the current state and a green CI run:
  the missing `conan/profiles/android`, an Android-compatibility audit
  of every native dep (the custom `icu-infinipaint` recipe is the
  biggest risk, mirroring the macOS debug cycle of v0.11.0-rc1..rc6),
  and the GitHub Actions job itself.
- The CI job is the small part. The recipe audit + cross-compile
  fight is the large part — realistically a week of debug cycles like
  the one we just did for macOS, more if Skia's Android path needs
  fresh work.
- Same soft-fail signing posture as desktop: ship unsigned APK when no
  release keystore is provisioned; only the `.aab` going to Play needs
  a real keystore.

## 1. What's already there

`android-project/` contains a near-complete SDL3 Android template:

| Path | What it is | State |
|---|---|---|
| `app/build.gradle` | AGP 8.7.3, compileSdk 35, minSdk 21, targetSdk 35, ndkVersion 29.0.14206865 | Configured, references missing profile |
| `app/src/main/java/com/erroratline0/infinipaint/InfiniPaint.java` | Main `SDLActivity` subclass | Pre-rebrand namespace |
| `app/src/main/java/org/libsdl/app/*` | Stock SDL3 Android Java (HID, controller, surface, input) | Stock — keep upstream-pristine |
| `app/src/main/AndroidManifest.xml` | App manifest | Needs rebrand audit |
| `gradlew`, `gradle/wrapper/*` | Gradle wrapper | Vendored |
| `settings.gradle`, `build.gradle` | Root + module | Standard |

`app/build.gradle` already runs `conan install ../../.. --profile android
-s build_type=$BT -s arch=$ARCH` for `armv8 × x86_64 × Debug × Release`
(four package builds), with abiFilters limited to `arm64-v8a` and
`x86_64` — the right shape for current devices (no 32-bit cruft).

The CMake bridge is already wired: `externalNativeBuild` points at
`../../CMakeLists.txt`, passes `-DCMAKE_TOOLCHAIN_FILE=conan_android_toolchain.cmake`
(generated by Conan via `tools.cmake.cmake_layout:build_folder_vars=['settings.arch']`),
and uses `-DGRAPHICS_BACKEND=OpenGLES3.0` to select the Android-friendly
Skia backend.

Assets are shared with desktop: `sourceSets.main { assets.srcDirs =
['../../assets'] }` — so `assets/data/config/default_p2p.json` (and the
TURN-secret injection step) reuse without duplication.

## 2. What's missing

### 2.1 The `android` Conan profile

`build.gradle:13` references `--profile android` but `conan/profiles/`
contains only `emscripten`, `linux-x86_64`, `macOS-arm64`, `win-x86_64`.

Sketch:

```ini
[settings]
os=Android
os.api_level=21
arch=armv8           # gradle overrides per-arch via -s arch=
compiler=clang
compiler.version=18
compiler.libcxx=c++_shared
compiler.cppstd=gnu23
build_type=Release

[tool_requires]
cmake/3.27.0

[conf]
tools.android:ndk_path=$ANDROID_NDK_HOME
tools.cmake.cmaketoolchain:user_toolchain=["{{ os.getenv('ANDROID_NDK_HOME') }}/build/cmake/android.toolchain.cmake"]
```

`compiler.version=18` matches NDK r27 / 29.0.x's bundled clang. Need to
confirm against the exact NDK we pin.

### 2.2 Native-dep Android compatibility audit

Every native Conan dep needs to actually build for `os=Android`. Risk
matrix:

| Dep | Android risk | Notes |
|---|---|---|
| **SDL3** | Low | First-class Android target upstream. The `android-project/` Java glue *is* the SDL Android template. |
| **libdatachannel** | Low–Medium | Has Android CMake paths, but our deps `plog` etc. have submodule clone fragility (see desktop rc6 fix). |
| **Skia** | **Medium–High** | Skia's Conan recipe and Android NDK don't always agree on flags. `-DGRAPHICS_BACKEND=OpenGLES3.0` already selects the right Android backend in our CMakeLists, but the Conan side may need extra options or a custom variant. |
| **`icu-infinipaint` (custom)** | **High** | Already proven hostile during macOS debug (v0.11.0-rc4 fix). The recipe's autotools path needs the same kind of careful flag-gating for `os=Android` (NDK clang isn't always happy with the autoconf cross-compile heuristics). Likely 1–2 fix cycles. |
| **libmypaint** | Medium | Pure C, CPU-bound — should compile. Custom recipe (`conan/libmypaint/`) needs `os=Android` not to hit `configure_file` paths that assume host tooling. |
| **freetype, harfbuzz, zlib, bzip2, etc.** | Low | conan-center has solid Android support for these. |
| **clip** (clipboard) | Medium | Desktop-clipboard library; Android needs the JNI variant or a stub. May be the right place to insert a no-op Android impl. |

The pattern from rc1..rc6: each broken recipe took one CI cycle to
surface, plus a small commit to fix. Plan for ~5–10 such cycles, give
or take.

### 2.3 The GitHub Actions job

A new `build-android` job in `.github/workflows/build-installers.yml`,
mirroring the existing platform jobs' shape:

```yaml
build-android:
  runs-on: ubuntu-22.04
  steps:
    - uses: actions/checkout@v4
      with: { submodules: recursive }
    - uses: actions/setup-java@v4
      with: { distribution: temurin, java-version: 17 }
    - uses: nttld/setup-ndk@v1
      id: ndk
      with: { ndk-version: r27c, link-to-sdk: true }
    - uses: android-actions/setup-android@v3
      with: { packages: 'platform-tools platforms;android-35 build-tools;35.0.0' }
    - uses: actions/setup-python@v5
      with: { python-version: '3.11' }
    - name: Install Conan
      run: pip install --upgrade "conan>=2.0,<3.0"
    - uses: actions/cache@v4
      with:
        path: ~/.conan2
        key: conan-android-${{ hashFiles('conanfile.py', 'conan/**/conanfile.py') }}
        restore-keys: conan-android-
    - name: Inject TURN secret
      env: { TURN_SECRET: ${{ secrets.TURN_SECRET }} }
      run: python scripts/inject_turn_secret.py
    - name: Gradle assemble
      env:
        ANDROID_NDK_HOME: ${{ steps.ndk.outputs.ndk-path }}
      working-directory: android-project
      run: |
        chmod +x ./gradlew
        ./gradlew assembleRelease bundleRelease
    - name: Sign APK / AAB (when keystore provisioned)
      if: ${{ env.HAVE_KEYSTORE == 'true' }}
      env: { ... apksigner config ... }
      run: |
        # apksigner sign --ks ... app-release-unsigned.apk
    - uses: actions/upload-artifact@v4
      with:
        name: android
        path: |
          android-project/app/build/outputs/apk/release/*.apk
          android-project/app/build/outputs/bundle/release/*.aab
        retention-days: 1
```

Plumb into `create-release` job's `needs: [..., build-android]` and add
APK/AAB to the artifact glob in the flatten step.

### 2.4 Signing strategy (soft-fail, mirroring desktop)

Same posture as macOS notarization and Windows signtool: **build always
succeeds; signing is opt-in based on secret presence**.

| Secret (proposed) | What | Status today |
|---|---|---|
| `ANDROID_KEYSTORE_B64` | Base64-encoded `.jks` for release signing | Not provisioned |
| `ANDROID_KEYSTORE_PASSWORD` | Keystore password | Not provisioned |
| `ANDROID_KEY_ALIAS` | Key alias inside the keystore | Not provisioned |
| `ANDROID_KEY_PASSWORD` | Per-key password | Not provisioned |

When the keystore secrets aren't set, the workflow ships an unsigned
APK marked `app-release-unsigned.apk` (same as the gradle default).
Side-loadable but Play Store will reject. Emit a `::notice::` to make
the unsigned status obvious in the run log.

Currently `app/build.gradle:57` sets `signingConfig
signingConfigs.debug` for the release variant — this means a release
APK *is* technically signed (with the debug keystore), which is fine
for testing but **not for distribution**. When the real keystore arrives
the `signingConfig` block needs to switch to a release variant.

## 3. Local Android build on this Windows machine

CI is one delivery path; a fast local loop is the other. Without a local
build, every Android iteration is an 8-15 minute CI round-trip — not
viable for the recipe-by-recipe audit §2.2 demands. This section is the
Windows 11 bring-up checklist for the local dev machine that's already
building desktop targets.

### 3.1 Toolchain install

Two routes:

**(a) Android Studio (recommended)** — bundles JDK 17, SDK,
sdkmanager, emulator, and a UI for managing NDK installs. After
install, open SDK Manager and pin:

- SDK Platform 35 — matches `compileSdk 35` in `app/build.gradle:31`
- NDK 29.0.14206865 — matches `ndkVersion` in `app/build.gradle:32`
- Build-Tools 35.0.0
- Platform-Tools (for `adb`)

**(b) Command-line only (faster, less disk)**:

```pwsh
winget install EclipseAdoptium.Temurin.17.JDK
# then download commandlinetools-win-*.zip from
# https://developer.android.com/studio#command-line-tools-only
# and extract to D:\android-sdk\cmdline-tools\latest\
sdkmanager "platform-tools" "platforms;android-35" "build-tools;35.0.0" "ndk;29.0.14206865"
```

### 3.2 Environment variables

```pwsh
setx ANDROID_HOME      "D:\android-sdk"
setx ANDROID_SDK_ROOT  "D:\android-sdk"
setx ANDROID_NDK_HOME  "D:\android-sdk\ndk\29.0.14206865"
setx JAVA_HOME         "C:\Program Files\Eclipse Adoptium\jdk-17.0.x-hotspot"
```

Append to PATH: `%ANDROID_HOME%\platform-tools` (`adb`) and
`%ANDROID_HOME%\cmdline-tools\latest\bin` (`sdkmanager`).

### 3.3 Windows-specific gotchas

These bite hard if not handled up front.

| Issue | Symptom | Fix |
|---|---|---|
| **Long paths** | NDK + Conan paths exceed 260 chars; recipe failures `path too long` | Enable Win32 long-path support (gpedit → Admin Templates → System → Filesystem → Enable Win32 long paths) and `git config --system core.longpaths true` |
| **Antivirus scans** | 5-10× slower; intermittent recipe fails | Defender exclusions for `D:\.conan2`, `D:\android-sdk`, `D:\repos\infinipaint` |
| **Symlink privileges** | `OSError: symbolic link privilege not held` mid-Conan-install | Enable Windows Developer Mode (Settings → Privacy → For developers) |
| **Gradle JVM heap** | `OutOfMemoryError` (Skia compile is heavy) | Add `org.gradle.jvmargs=-Xmx6g -XX:MaxMetaspaceSize=1g` to `android-project/gradle.properties` |

### 3.4 Author the Conan profile

§2.1's `conan/profiles/android` doesn't exist yet — authoring it is
the first concrete local step. Use the §2.1 sketch as a starting
point; expect to revise as recipe fixes land.

### 3.5 Exercise Conan install before gradle

Faster failure surface than going through gradle:

```pwsh
uv run --no-project conan install . --profile android -s arch=armv8 -s build_type=Release --build=missing
```

Each failure is one row of the §2.2 risk matrix made concrete. Fix
the recipe, re-run. When `armv8` is clean, repeat for `x86_64`.

### 3.6 First gradle build

```pwsh
cd android-project
.\gradlew.bat assembleDebug
```

Output APK: `app\build\outputs\apk\debug\app-debug.apk`. First run
adds gradle dependency resolution + cmake configure (~3-5 min);
subsequent builds are incremental.

### 3.7 Deploy + run

**Physical device** (recommended — touch input is the actual unknown):

1. Settings → About → tap "Build number" 7× to enable Developer Mode
2. Settings → Developer Options → enable USB debugging
3. Plug in via USB; accept the fingerprint prompt
4. `adb devices` should list the phone
5. `adb install -r app\build\outputs\apk\debug\app-debug.apk`
6. `adb shell am start -n com.erroratline0.InfiniPaint/.InfiniPaint`
   (pre-rebrand applicationId — rename per §6/A5)

**Emulator** (slower; useful for verifying x86_64 builds):

```pwsh
sdkmanager "system-images;android-35;google_apis;x86_64"
avdmanager create avd -n inkternity_test -k "system-images;android-35;google_apis;x86_64"
emulator @inkternity_test
```

### 3.8 Iteration loop

```pwsh
cd android-project
.\gradlew.bat installDebug              # build + install in one shot
adb logcat -s SDL,Inkternity:V *:S      # filtered runtime logs
```

`adb logcat` filtered to SDL + Inkternity tags is essential for
debugging touch input, lifecycle transitions, and SDL_Log output from
the C++ side.

## 4. Build outputs

For one tagged release, the Android job produces:

- `app-release.apk` (or `app-release-unsigned.apk`) for direct sideload
- `app-release.aab` (Android App Bundle) for Play Store upload

Both go into the GitHub Release assets alongside the desktop installers.
APK is what most testers actually want; AAB is only useful when there's
a Play Console listing to upload it to.

If the AAB output adds noise without benefit at this stage, drop it
from the workflow until Play distribution is on the roadmap. Easy
one-liner to add later.

## 5. Open questions

1. **Skia GPU backend on Android.** Currently `-DGRAPHICS_BACKEND=OpenGLES3.0`.
   Is that what we want long-term, or should we be targeting Vulkan
   via Skia's `SkiaSurface` for newer devices? GLES 3.0 is the safe
   minimum (covers basically every device since 2014). Vulkan would
   require Android API 26+ effectively.

2. **Touch input.** Inkternity's input layer is built around mouse +
   stylus pressure events. Android touch events go through SDL3's
   touch handling — does the existing pressure-curve / brush-engine
   code path consume `SDL_FINGERDOWN/MOTION/UP` correctly, or is it
   only wired for `SDL_MOUSE*`? Untested. A second day of work if it's
   wrong.

3. **Mobile UI density.** The desktop UI is tuned for keyboard +
   mouse. Touch targets, gesture handling (pinch-to-zoom, two-finger
   pan), and on-screen toolbars need rework. The Phase 0 distribution
   model is reader-as-subscriber, so launching with viewer-only on
   Android first is plausible (no editing UI needed).

4. **Distribution channel.** Play Store requires AAB + signed by the
   real keystore + a developer account ($25 one-time) + Play Console
   listing. F-Droid is a possible alternative for an open-source
   Android target — no fee but tighter rules around bundled binaries.
   Side-loaded APK is the path of least resistance for testing.

5. **Time bounding.** This is meaningful scope creep relative to
   Phase 0 (desktop-only by design). Do we Phase-0.5 it (after desktop
   v0.11.0 ships and the signaling/TURN infrastructure has soaked),
   or fold it into Phase 0 and slip the v0.11.0 ship date by 1–2
   weeks?

## 6. Recommended sequencing

If the answer to (5) above is "do it after desktop ships":

1. Land desktop v0.11.0 first (B11). Get the signaling + TURN infra
   battle-tested with real subscribers.
2. **A1**: author `conan/profiles/android`, run `conan install
   ../../.. --profile android -s arch=armv8 --build=missing` locally,
   capture every recipe failure, fix them one at a time.
3. **A2**: get a clean local `./gradlew assembleDebug` against the
   armv8 ABI from a Linux dev box.
4. **A3**: add the `build-android` job to the workflow, debug in CI
   (expect a similar rc-tag cycle to the macOS work).
5. **A4**: provision the release keystore; switch from soft-fail
   unsigned to soft-fail signed.
6. **A5**: rebrand pass — namespace, applicationId, AndroidManifest
   labels, app icon, splash screen.
7. **A6**: real-device smoke test (touch input, lifecycle, asset
   bundling, file system paths).
8. **A7**: first real Android release in the desktop release flow.

If the answer is "fold into Phase 0":

Same sequence but A1–A3 happen in parallel with B7–B11 instead of
after them. Higher coordination cost; risks slipping desktop ship.
